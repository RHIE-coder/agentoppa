# 견본 — 두루뭉술한 요청에서 의도 정리까지

두루뭉술한 한 줄에서 시작해 넓혀 떠보며 좁히고, 되짚어 확인으로 확정해 `.harness/<하네스이름>/intent.md`를 남기는 한 판. 형식·톤·길이의 목표로 삼는다. 아래는 **프로젝트 수준** 면담(이 프로젝트를 흐름에 맞추는 자리) 한 판이고, 끝에 **프레임워크 수준**(재사용 Core 짓기) 한 판을 짧게 덧붙인다.

## 프로젝트 수준 면담 (한 번에 한 질문)

**사용자:** "내 프로젝트에 AI 작업 흐름 좀 만들어줘. 그냥 알아서 잘 되게."

> 두루뭉술·"알아서" → **넓히기**. 열린 "뭘 원하세요?" 대신 양 극단 두 개.

**면담:** "두 그림 중 어디에 가까워요? ⓐ 코드 짜기 전에 기획·명세를 탄탄히 잡는 흐름, ⓑ 일단 만들고 리뷰·QA로 품질을 조이는 흐름."

**사용자:** "음… ⓑ 쪽인데, 리뷰는 자동이면 좋겠고."

> 방향이 잡힘 → 임시 안을 던져 좁힌다.

**면담:** "그럼 기본안: 구현하면 자동으로 리뷰가 돌고, 통과 못 하면 되돌린다. 리뷰가 보안까지 봐야 해요, 아니면 정확성·가독성이면 충분?"

**사용자:** "보안도. 로그인 건드리는 데가 있어서."

**면담:** "리뷰에서 심각한 문제가 남으면 진행을 *막을까요*, 경고만 하고 진행은 둘까요?"

**사용자:** "막아줘. 대충 넘어가는 게 싫어."

> 핵심이 다 잡힘 → **되짚어 확인**.

**면담:** "정리하면 — 구현 후 코드+보안 리뷰를 자동으로 돌리고, 심각한 문제가 남으면 진행을 막음. 기획은 안 넣음. 맞나요?"

**사용자:** "응 딱 그거야."

## 결과: `.harness/<하네스이름>/intent.md`

```markdown
---
phase:  intent-interview
status: ready
inputs: []
level:  project
---
# Intent: 구현 후 자동 리뷰로 품질을 조이는 흐름 (보안 포함, 심각한 문제는 막기)

## 목표
- 손으로 일일이 안 챙겨도 바뀔 때마다 품질·보안이 강제되는 개발 흐름.

## 범위
- 할 것(in): 구현, 자동 코드 리뷰, 자동 보안 리뷰, 통과 게이트.
- 안 할 것(out): 사전 기획·명세, 화면 다듬기.

## 제약
- 로그인/인증 코드가 있어 보안 리뷰 필수.

## 예시 / 반례
- 싫은 사례: 심각한 지적이 있는데 그냥 다음으로 넘어가는 것.

## 우선순위
1. 심각한 문제는 반드시 막기 (대충 통과 금지).
2. 리뷰 자동화.

## 내린 결정
- 기획 우선 vs 사후 품질 → 사후 품질 (사용자가 "일단 만들고 조이기" 선호).
- 리뷰 범위 → 정확성+보안 (로그인 때문).
- 게이트 세기 → strict (심각한 문제 막기).

## 미해결
- 차단: 없음
- 비차단: 테스트 명령·CI 연동은 만들 때 확인.

## 확신
- 필수 항목 구체 답: 예
- 구체 예시 확보: 예 (싫은 사례)
- 결과 가르는 미지수 없음: 예
- 마지막 되짚어 확인에 정정 없음: 예
- 한 줄 테스트: 예 — 만드는 사람이 기획 없이 리뷰-게이트 개발 흐름을 만든다.

## 하네스 힌트
- 작업 묶음: 구현 + review (code-reviewer, security-reviewer)
- 분야: 소프트웨어 개발
- sync: strict
- routing(모델 등급): balanced
```

## 왜 좋은 견본인가
- **한 번에 한 질문** — 매 번 하나씩. 사용자 답이 다음 질문을 정한다.
- **두루뭉술 → 넓히기** — "뭘 원하세요?" 대신 양 극단·임시 안으로 *고르게* 했다.
- **되짚어 확인으로 확정** — 넘기기 전 그림을 되읽어 "딱 그거"를 받았다(확신).
- **하네스 힌트** — agent-engineer가 바로 `[review]`+strict+software-dev로 짤 수 있게 의도를 옮겼다.

---

## 프레임워크 수준 면담 (재사용 Core 짓기) — 같은 방법, 다른 축

방법(한 번에 한 질문·넓히기/좁히기)은 똑같다. *무엇을 캐는지*만 다르다 — 단계 흐름·**어디를 빈자리로 뺄지**·겨냥 프로젝트군.

**사용자:** "우리 팀 여러 레포에 똑같이 쓸 표준 작업 흐름을 만들고 싶어."

> "여러 레포에 두고두고" → **프레임워크 수준**(재사용 Core). 무엇을 *안* 박을지가 핵심.

**면담:** "그 흐름이 *반드시* 강제하길 원하는 한 가지는요? 예: 리뷰 없인 머지 금지, 또는 테스트 통과 없인 다음 단계 금지."

**사용자:** "테스트 통과 없인 다음 단계 못 가게."

**면담:** "그런데 레포마다 테스트 *도는 법*이 다르죠 — 어디는 `go test`, 어디는 `npm test`. 그 '테스트 명령'을 Core가 박지 말고 *빈자리*로 둬서 레포마다 채우게 할까요?"

**사용자:** "응, 그래야 재사용되지. e2e도 마찬가지로 빈자리."

**면담:** "겨냥하는 레포가 web·go 백엔드 둘 다인가요, 아니면 더 넓게 도메인 안 가리게?"

**사용자:** "지금은 web·go 둘. 나중에 더 붙을 수도."

> 핵심이 잡힘 → **되짚어 확인.**

**면담:** "정리하면 — 구현→테스트→리뷰 흐름, 테스트 통과 전엔 다음 단계 막기(strict). `test-command`·`e2e-runner`는 Core가 안 박고 빈자리로 두어 레포마다 바인딩. web·go 겨냥. 맞나요?"

**사용자:** "딱 그거."

### 결과: `.harness/<하네스이름>/intent.md` (level: framework)

```markdown
---
phase:  intent-interview
status: ready
inputs: []
level:  framework
---
# Intent: 팀 표준 "테스트 통과 게이트" 재사용 Core (web·go 겨냥)

## 목표
- 어느 레포에 얹어도 테스트 통과 없인 다음 단계로 못 가게.

## 범위
- 할 것(in): 구현→테스트→리뷰 흐름, 게이트, test/e2e 빈자리.
- 안 할 것(out): 레포별 테스트 명령을 Core에 박기 (재사용 깨짐).

## 제약
- web·go 둘 다 통해야 함 → 도메인 값 안 박기.

## 우선순위
1. 재사용성(값 안 박기) — 빈자리로 빼는 게 최우선.
2. 테스트 게이트 강제.

## 내린 결정
- 강제할 규율 → 테스트 통과 전 진행 금지(strict).
- test-command·e2e-runner → 빈자리(requires)로 (레포마다 다름).

## 미해결
- 차단: 없음
- 비차단: 린터도 빈자리로 뺄지는 첫 프로젝트 바인딩 때 확인.

## 확신
- 필수 항목 구체 답: 예
- 구체 예시 확보: 예 (go test vs npm test)
- 결과 가르는 미지수 없음: 예
- 마지막 되짚어 확인에 정정 없음: 예
- 한 줄 테스트: 예 — 만드는 사람이 값 안 박은 재사용 흐름을 짓는다.

## Core 설계 힌트
- 철학(강제할 규율): 테스트 통과 없인 다음 단계 금지.
- 작업 단계 묶음 + sync: 구현 → 테스트 → 리뷰, strict.
- 인터페이스 빈자리(requires): test-command, e2e-runner.
- 겨냥 프로젝트군: web·go 백엔드 (도메인 안 박음).
```

**두 견본의 차이 한 줄:** 프로젝트 수준은 *이 레포*의 목표·바인딩을 캐고(→`config.yaml`), 프레임워크 수준은 *흐름과 빈자리*를 캔다(→재사용 Core). agent-engineer가 `level`로 갈라 받는다.
