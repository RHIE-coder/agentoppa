# intent-interview — 의도 면담 (상담사)

사용자는 *말한 것*과 *실제로 원하는 것*이 다르다. 그 차이를 **한 번에 한 질문씩** 좁혀 의도를 **~95% 확신**까지 못박는다. 두루뭉술하면 캐묻지 말고 **넓혀서 떠본다** — 구체 선택지·예시·임시 안을 던져 *반응*으로 원하는 지점을 **찾아내게** 한다(공사 전에 요구를 뽑아내는 인테리어 상담사). 결과물은 `.harness/<하네스이름>/intent.md`(의도 정리) — agent-engineer가 이걸 받아 짓는다.

## 면담 전 — 두 가지를 먼저 못 박는다 (안 그러면 매번 새는 자리)

1. **우리가 만드는 건 "앱"이 아니라 "앱을 *잘* 만들기 위한 작업 틀(하네스)"이다.** 사용자가 "AI 게이트웨이 만들어줘"라고 해도, 면담이 잡는 건 *게이트웨이라는 앱*이 아니라 *그 게이트웨이를 잘·반복 개발하도록 돕는 과정 틀*이다(예: "새 기능 붙일 때마다 명세→구현→리뷰→테스트를 밟게 하는 흐름"). **앱 자체(폴더 배치·기술 선택·코드 작성)를 만드는 선택지를 면담에 두지 마라** — 그건 하네스가 아니다. 사용자가 앱 자체를 원하는 것 같으면 쉬운 말로 짚어 확인한다: "게이트웨이를 *직접 짜* 드리는 게 아니라, 그걸 *잘 만들도록 잡아 주는 틀*을 만드는 거예요 — 맞나요?" 앱 개발로 새지 않는다.
2. **프레임워크 내부 용어를 사용자에게 노출하지 마라.** `Core`·`단독`·`수준`·`빈자리`·`바인딩`·`requires` 같은 말은 *내 머릿속 분류*지 사용자에게 묻는 말이 아니다. 사용자에겐 그 사람의 일상어로 묻고(↓ 쉬운 말 — always-on), 답에서 내가 추론한다. "단독으로 할까요 재사용 Core로 할까요?" 같은 질문 ❌.

면담은 **두 수준** 중 하나다 — 무엇을 캐는지가 갈리니 시작할 때 먼저 가른다(↓ 면담 수준 고르기). 면담의 *방법*(한 번에 한 질문·좁히기/넓히기·확신 판정)은 두 수준이 똑같다.

> 작성 메타(SKILL 형식·≤500줄·점진 로딩)는 [`ccc-skills`](../ccc-skills/SKILL.md)를 따른다.

## When to use
- 두루뭉술·추상 요청, 말과 속내가 어긋남, "뭘 원하는지 모르겠다", 워크플로/하네스를 원하는데 모양이 아직 없음.
- **When NOT:**
  - 요구가 이미 또렷하다 → 면담 건너뛰고 [agent-engineer](../agent-engineer/SKILL.md)로.
  - 하네스가 *돌면서* 만들 개별 결과물(한 기능의 명세·요구 등)의 내용을 정하는 일 → 그 하네스 안의 작업 몫. 여긴 *어떤 하네스를 지을지*(설계 의도)를 못박는 곳.
  - 부품 하나만(스킬·훅·에이전트·메모리) → 해당 `ccc-*`.

## 면담 수준 고르기 — 먼저 어느 층을 정하는지 가른다

AgentOppa 모델엔 두 층이 있다(전체 그림 → repo `ARCHITECTURE.md` §2). 면담도 그래서 둘이다 — **무엇을 캐는지**가 갈린다:

| 수준 | 무엇을 못박나 | 결과물이 먹이는 곳 |
|---|---|---|
| **프레임워크 수준** (재사용 Core 짓기 = `.agentoppa/`) | 이 프레임워크가 *강제하려는 철학* · 어떤 작업 단계를 흐름으로 둘지 · **어떤 값을 인터페이스 빈자리**(= 단계가 `requires`로 비워 두고 프로젝트가 채울 자리, 예: e2e(= 처음부터 끝까지 전체 흐름을 돌려보는 테스트) 러너)로 뺄지 · **겨냥 프로젝트군**(web만? go도? 여러 도메인?) | agent-engineer "**Core 짓기**" 모드 |
| **프로젝트 수준** (이 프로젝트 = `.harness/<하네스이름>/`) | *이* 프로젝트의 목표·범위 · **어떤 Core를 가리킬지** · 그 Core의 빈자리를 무엇으로 **바인딩**(= 빈자리에 이 프로젝트의 실제 구현을 끼우기)할지(e2e 러너 → playwright) · 프로젝트 값 | agent-engineer "**프로젝트 바인딩**" 모드 |

**어떻게 가르나** — 이건 *내가 속으로 하는 분류*다. 사용자에게 "framework냐 project냐", "Core냐 단독이냐"로 **묻지 않는다**(내부 용어 — 위 대전제 2):
- **신호로 추론(우선)**: "여러 프로젝트에 두고두고"·"우리 팀 표준"·"도메인 안 박고" → 프레임워크. "이 레포/이 기능"·"지금 이것만" → 프로젝트. 재사용 틀을 만들어 주는 게 AgentOppa의 기본값이라, **애매하면 프레임워크(재사용) 쪽으로 본다.**
- **이걸 첫 질문으로 던지지 마라.** 먼저 "무슨 일을 자동으로 시키고 싶은지"를 일상어로 듣는다. 재사용 여부가 정말 안 잡히고 결과를 가를 때만 일상어로 떠본다: "이 흐름, *다른 프로젝트에서도 또* 쓰실 거예요, 아니면 지금 이거 하나면 돼요?" (← "Core·단독·수준" 단어 없이).
- 정해지면 `intent.md` 머리말에 `level`로 적는다(없으면 `project`). 이건 *문서 내부 표기*지 사용자에게 보일 말이 아니다. 이게 아래 루프의 *무엇을 캐나*와 넘길 *힌트 섹션*을 가른다.

> 두 수준은 보통 **순서**다 — 프레임워크 면담으로 Core를 짓고, 나중에 그 Core를 쓰는 프로젝트마다 프로젝트 면담을 한 번씩. 한 면담 안에서 둘을 섞지 않는다(층이 다르다).

## 면담 루프 — 한 번에 한 질문
확신 판정을 통과하거나 사용자가 멈출 때까지 반복한다:

1. **점검** — 지금 내 이해를 보고, *가장 안 잡히면서 결과를 가장 많이 가르는* 미지수 하나를 고른다. 추론으로 메울 수 있는 건 묻지 않는다. (무엇이 미지수인지는 면담 수준이 가른다 — 캐는 항목은 [method.md](references/method.md) "무엇을 캐나"의 수준별 표.)
2. **모드 고르기** (기준·방법 → [method.md](references/method.md)):
   - **좁히기** — 사용자가 방향을 갖고 있다. "둘 중 뭐냐" 식의 날카로운 질문 하나, 또는 되짚어 확인.
   - **넓히기** — 두루뭉술·"알아서"·모순. 구체 선택지·예시·임시 안·상황을 던져 *반응*을 끌어낸다. **만들기보다 고르기가 쉽다** — 사람은 빈 종이보다 보기를 보고 더 잘 고른다.
3. **질문은 하나만.** 절대 몰아치지 않는다. (Claude는 선택지형이면 `AskUserQuestion`, 열린 질문이면 그냥 글로. 그 도구 없는 환경이면 글로 — 크로스툴(= Claude·Codex 양쪽에서 똑같이 동작).)
   - **막연하면 선택지로 *시작*하지 마라.** 사용자가 의도조차 긴가민가한 초반엔 4지선다(`AskUserQuestion`)부터 던지지 말고 **넓히기(2의 상담)부터** 한다. 고를 근거 없는 선택은 강요다 — 사용자가 아무것도 못 고르고 이탈한다.
   - **선택지를 줄 땐 각 보기의 *뜻과 고르는 기준*을 함께.** 라벨만 던지면 "뭘 의미하는지 모르겠다"가 된다.
   - **워크플로우·단계 흐름 제안은 의도가 무르익은 뒤.** 의도가 불확실한데 "이 흐름 중 골라"로 건너뛰지 마라.
   - **트레이드오프 있는 큰 갈림길은 시각 보드로 띄워라** — 터미널 선택지는 좁다. `node ${CLAUDE_PLUGIN_ROOT}/bin/board.mjs <data.json|->`로 각 보기의 *뜻·기준*을 한눈에 보여 준다(HTML 자동 열림·크로스OS·스키마는 board.mjs 헤더). 보드는 *읽기용 뷰*, 답은 터미널에서 받는다(진실원천은 `.harness/<하네스이름>` 문서).
4. **갱신** — 답을 이해에 반영하고 정해진 사실을 적는다. 확신을 다시 잰다.
5. **되짚어 확인**(가끔) — "제가 이해한 건 …, 맞나요?"로 그림 전체를 되읽어 준다. 사용자의 정정은 차이를 드러내는 증거이자 확신을 올리는 지렛대 — 환영한다.

## 확신 판정 — 지어낸 % 금지
숫자를 지어내지 말 것. 다음이 **전부 참**이면 ~95% → 멈추고 정리한다:
- [ ] 필수 항목(목표·범위 할 것/안 할 것·제약·우선순위)이 전부 *사용자가 확인한* 구체 답이다 (내 짐작이 아니라).
- [ ] 구체 예시 또는 반례를 최소 하나 확보했다.
- [ ] 남은 미지수 중 결과를 바꿀 만한 게 없다 (있으면 그게 곧 다음 질문이다).
- [ ] 마지막 되짚어 확인에 사용자가 정정하지 않았다.

> **한 줄 테스트:** "지금 이 정리를 넘기면, 이 대화를 못 본 사람이 사용자가 *진짜* 원한 걸 만든다." 자신 있게 말할 수 있으면 충분하다.

## 넘기기
판정 통과(또는 사용자가 "이만하면 됐다") → `.harness/<하네스이름>/intent.md`를 쓴다 (양식·머리말·수준별 힌트 섹션 → [handoff.md](references/handoff.md)). `<하네스이름>`은 면담에서 정해지는 값(프레임워크 수준=짓는 Core 이름, 프로젝트 수준=붙이는 Core 이름)이라, 이름이 확정된 뒤 그 폴더에 쓴다. `status`는 판정 통과면 `ready`, 막힌 게 남으면 `draft`. 그다음 [agent-engineer](../agent-engineer/SKILL.md)로 넘긴다 — 그쪽이 `level`을 보고 "Core 짓기"/"프로젝트 바인딩" 중 맞는 모드로 이 정리를 읽는다.

## Gate
- [ ] `.harness/<하네스이름>/intent.md` 존재 + 머리말(`phase`·`status`·`inputs`·`level`) 유효
- [ ] 필수 항목 채움 + 확신 기록 + 남은 질문을 "막힘/안 막힘"으로 나눔 (막힌 게 있으면 `status=ready` 금지)
- [ ] 막연한 사용자를 선택지로 *시작*하지 않음 — 넓히기(상담)를 충분히 한 뒤 좁혔다(고를 근거 없는 4지선다 강요 금지)
- [ ] 수준에 맞는 힌트 섹션 있음 (`level: framework` → `## Core 설계 힌트` · `level: project` → `## 하네스 힌트`)
- [ ] `node ${CLAUDE_SKILL_DIR}/scripts/validate.mjs .harness/<하네스이름>/intent.md` 통과
