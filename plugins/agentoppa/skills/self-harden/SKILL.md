# self-harden — 정정을 영구 가드로 (쓸수록 똑똑해지는 루프)

한 번 정정한 실수는 *다음 세션이 똑같이 반복*한다 — 교훈이 **그 일을 하는 순간 저절로 떠오르지** 않으면. self-harden은 정정을 **가장 강한 장치**에 주입해 시스템이 *스스로 단단해지게* 한다. 두 가지만 기억:

- **모아두지 말고 알맞은 곳에 주입한다.** 자꾸 쌓이는 메모(`mistakes.md` 같은)는 컨텍스트만 잡아먹고 정작 그 순간엔 안 떠오른다. 교훈은 *그게 필요한 자리*에 꽂는다.
- **맹점은 자동 검사로 막는다.** *맹점* = 내가 틀린 줄도 모르는 실수. 모르니까 찾아볼 수조차 없다 → 내가 안 찾아도 그 순간 알아서 튀어나오는 검사로 주입해야 막힌다. 읽어야 보이는 문서는 그 다음이다.

> 작성 메타(SKILL 형식·≤500줄·점진 로딩)는 [`ccc-skills`](../ccc-skills/SKILL.md)를 따른다. 출력은 `always-on.md`의 *쉬운 말* 규칙대로 — 처음 보는 사람도 읽고 무슨 일을 하는지 알게.

## When to use
- 사용자가 접근을 정정함, 같은 종류 실수가 또 날 낌새, "다신 이러지 마 / 다음 세션도 안 틀리게 / 굳혀줘".
- **When NOT:**
  - 또 날 일 없는 *한 번뿐인* 버그 → 그냥 고치고 끝. self-harden은 *같은 종류 전체*를 막는 일.
  - 새 기능·리팩터 그 자체 → 해당 작업 스킬로.

## 루프
정정 하나를 영구 가드로 바꾼다:

1. **포착** — 정정을 `.agentoppa/candidates/`에 후보로 싸게 적어둔다 ([template.md](template.md) 양식).
2. **일반화** — *이번 한 번*이 아니라 *같은 종류 전체*로. 고유명사(특정 파일·이름)를 떼고 부류로 올린다. → 이 한 줄이 **제목**이 된다.
3. **수단 판정** — 강하고 싼 것부터 *순차로* "이 교훈을 이 수단으로 잡나?" 따져 최상위에 배정(쪼갬 가능). 절차·각 수단 설명 → [classify.md](references/classify.md).
4. **범위 + 자리** — 배포(유저에도)냐 내부(이 repo만)냐 → 자리가 갈린다(아래 "배치 구조"). 실제 넣는 법 → [route.md](references/route.md).
5. **확인 게이트** — 주입 전, 아래 *화면*으로 보고하고 확인받는다. **틀린 교훈은 안 주입하느니만 못하다.**
6. **주입 + 기록** — 확인되면 (a) **산출물**(범용 가드)을 자리에 주입, (b) **로그**(이번 사건·추론)를 `.agentoppa/hardening-log.md`에 남긴다. 후보는 비운다.

## 산출물 vs 로그 — 반드시 분리
- **산출물(가드)** = *범용·일반*만. 라이브러리처럼 — **특정 사건 이름(인스턴스)을 넣지 않는다.**
- **로그** = `.agentoppa/hardening-log.md`(append-only = 뒤에만 덧붙이고 기존 줄은 지우거나 고치지 않음). *무슨 일·왜 생겼나·어떻게 일반화·왜 이 수단·기대*. 인스턴스 이름은 **여기에만**.
- 이유: 산출물이 깨끗해야 재사용되고, 경위는 다음 사람이 읽고 잇는다(project-committed).

## 확인 게이트 — 출력 화면 (쉬운 말)
제목 = **그 교훈의 한 줄 요약**(2단계 일반화). ⚠ 도구 정의("실수 안 하게")나 사용자가 친 말을 그대로 쓰지 않는다 — 자동 요약한다.

```
🔧 self-harden · <교훈 한 줄 요약>
   <한 문장 풀이 — 이번에 굳히는 것>

[1] 무슨 일이 있었나     <포착: 사건>
[2] 같은 종류 전체로     <일반화: 부류 — 인스턴스 이름 뺌>
[3] 어떻게 막을까        <수단 순차 판정 — 각 수단이 뭐고/이번에 맞는지, ✓ 채택>
[4] 누구에게 적용        <범위: 배포 | 내부 + 한 줄 근거>
[5] 어디에 넣나          산출물 → <경로>(범용) · 로그 → .agentoppa/hardening-log.md
[6] 로그 전문            <hardening-log.md에 남길 전말>
[7] 이대로 진행할까요?   [진행] / [고칠 점]
```

## 배치 구조 — 범위가 자리를 정한다
| 범위 | 어디에 | 규칙 / 검사기 / 훅 |
|---|---|---|
| **배포** (유저에도 + 우리 dev엔 `--plugin-dir`로) | `plugins/agentoppa/` | `always-on.md` / `skills/*/scripts/validate.mjs` / `hooks/` |
| **내부** (이 repo만, ship 안 됨 = 유저에게 배포 안 됨) | repo 루트 | `AGENTS.md` / `test/` / (git 훅 등) |

(어느 쪽이든 **로그**는 `.agentoppa/hardening-log.md` — 우리 기록이라 ship 안 함.)

## 수단 한눈에 — 강하고 싼 것부터 (절차·설명 → [classify.md](references/classify.md))
| 수단 | 무엇 / 효과 | 평소 컨텍스트 |
|---|---|---|
| 검사기(+빨강/초록) | 코드가 자동 검사해 어기면 실패 (제일 셈) | **0** |
| 훅 | 특정 행동 순간 자동 실행 | **0** |
| 리뷰어 한 줄 | 리뷰 담당 에이전트에 한 줄 | 리뷰 안에서만 |
| Gate 한 줄 | 단계 '끝' 전 통과 조건 | 그 스킬 쓸 때 |
| 문서 한 줄 | 그 작업 설명서에 한 줄(읽을 때만) | 그 작업 때만 |
| 항상 규칙 | 모든 세션에 늘 뜸 (한 줌만) | 항상 |

## Gate
- [ ] 후보가 *같은 종류 전체*로 일반화됨 (인스턴스 이름은 산출물에 안 들어감)
- [ ] 수단을 *강하고 싼 것부터 순차 판정*해 배정 (자동으로 잡히는데 문서로 미루지 않음)
- [ ] 범위(배포/내부) → 자리 정해짐
- [ ] 기계로 잡는 거면 **빨강/초록 테스트 한 쌍**이 룰과 함께
- [ ] 확인 게이트(화면) 통과 + 산출물 주입 + `hardening-log.md` 기록 + 후보 비움
